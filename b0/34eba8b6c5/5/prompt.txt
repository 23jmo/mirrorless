### B. Direct Retailer Scraping

**How it works:** You build or use scrapers that visit individual retailer websites (Zara, H&M, ASOS, Nordstrom, etc.) and extract product data from their HTML or internal APIs.

**Approaches:**
- **HTTP-based scraping** (requests + BeautifulSoup/lxml): Works for server-rendered pages. Fast, lightweight, low cost.
- **Internal/undocumented APIs**: Many retailers load product data via XHR/fetch requests to internal JSON APIs. Reverse-engineering these yields clean, structured data without HTML parsing. ASOS, H&M, and Zara all use internal APIs heavily.
- **Browser automation** (Playwright/Puppeteer): Required for heavily JS-rendered sites. More resource-intensive.

**Retailer-specific notes:**

### E. Browser Automation (Puppeteer / Playwright)

**How it works:** A headless or headed browser (Chromium, Firefox, WebKit) is programmatically controlled to navigate sites, wait for JS rendering, and extract data from the fully rendered DOM.

**Tools:**
- **Playwright** (Microsoft): Supports Chromium, Firefox, WebKit. Python, Node.js, Java, .NET. Currently the preferred choice for new projects.
- **Puppeteer** (Google): Chromium/Chrome only. Node.js. Mature ecosystem.
- **Selenium:** Older, slower, but widest language support. Less recommended for new projects.
- **Crawlee** (Apify): Open-source framework built on Playwright/Puppeteer with built-in anti-detection, proxy rotation, session management, and autoscaling.

**When required:**
- Single-page applications (React/Vue/Angular stores like Zara, Nike)
- Sites that load products via infinite scroll
- Sites requiring user interactions (clicking size selectors, color swatches)
- Sites with heavy client-side rendering

**Anti-detection add-ons:**
- `playwright-stealth` / `puppeteer-extra-plugin-stealth`: Patches browser fingerprints
- Custom user-agent rotation
- Realistic mouse movements and scrolling
- Cookie and session management

**Pros:**
- Can handle any website regardless of JS complexity
- Sees exactly what a human browser sees
- Can interact with elements (dropdowns, filters, pagination)

**Cons:**
- Resource-intensive (each browser instance uses ~100-300MB RAM)
- Slower than HTTP-based scraping (5-10x)
- Higher proxy costs due to bandwidth
- Still detectable by advanced anti-bot systems
- Complex to scale

## 3. Legal Considerations

### Key Legal Frameworks

**Computer Fraud and Abuse Act (CFAA) -- United States:**
- The landmark **hiQ Labs v. LinkedIn** (2022) decision established that scraping publicly available data does not violate the CFAA. The Supreme Court declined to hear LinkedIn's appeal.
- However, this applies to *publicly accessible* data. Scraping behind a login wall (requiring credentials) could still violate the CFAA.
- The CFAA prohibits "unauthorized access" to computer systems. Courts have increasingly interpreted this narrowly for public websites.

**Terms of Service (ToS):**
- Most retailers explicitly prohibit scraping in their ToS
- Violating ToS is generally a breach of contract, not a criminal matter
- However, ToS violations can lead to cease-and-desist letters, IP bans, and civil lawsuits
- The enforceability of "browsewrap" ToS (where you don't explicitly agree) is debatable

**robots.txt:**
- A voluntary standard; not legally binding in the US
- Ignoring robots.txt can be used as evidence of bad faith in legal proceedings
- Best practice: always respect robots.txt directives
- Some sites use robots.txt aggressively to block all bots

**GDPR (EU) and Privacy Laws:**
- Scraping personal data (reviewer names, user profiles) in the EU triggers GDPR compliance requirements
- Product data (prices, descriptions) is generally not personal data
- California's CCPA has similar protections for personal information

**Copyright:**
- Product images are copyrighted by the retailer/brand
- Product descriptions may be copyrighted
- Factual data (prices, sizes, colors) is generally not copyrightable
- Using scraped images in your own app/site without permission is risky

### Practical Legal Guidance

1. **Safest:** Use official APIs (Amazon PA-API, eBay API) or licensed data providers (EDITED, Bright Data datasets)
2. **Moderate risk:** Scrape publicly available factual data (prices, availability) while respecting robots.txt and rate limits
3. **Higher risk:** Scraping and republishing copyrighted content (images, descriptions), scraping behind login walls, ignoring cease-and-desist requests
4. **Recommendations:**
   - Consult a lawyer if scraping is core to your business
   - Keep scraping frequency reasonable (don't DDoS the site)
   - Respect robots.txt
   - Don't circumvent technical access controls
   - Don't scrape personal data without a legal basis
   - Attribute data sources where appropriate
   - Be prepared to stop scraping a site if you receive a C&D

## 4. Technical Challenges

### Anti-Bot Measures by Difficulty

**Tier 1 -- Basic (easy to bypass):**
- User-agent checking
- IP-based rate limiting
- Basic referrer checking
- Simple cookie requirements

**Tier 2 -- Intermediate:**
- **Cloudflare** (used by Zara, many DTC brands): JavaScript challenges, browser fingerprinting, turnstile CAPTCHAs
- **Akamai Bot Manager** (used by Nordstrom, Nike, many large retailers): Device fingerprinting, behavioral analysis, session validation
- **DataDome** (growing adoption): ML-based bot detection, real-time behavioral analysis
- Rate limiting with sliding windows
- Dynamic CSS class names (changes with each deploy)

**Tier 3 -- Advanced (difficult):**
- **PerimeterX (now HUMAN)**: Advanced fingerprinting, proof-of-work challenges
- Canvas fingerprinting
- WebGL fingerprinting
- TLS fingerprinting (JA3/JA4)
- Behavioral analysis (mouse movements, scroll patterns, typing cadence)
- Honeypot links/forms
- Dynamic rendering that serves different content to bots

### Content Loading Challenges

- **Infinite scroll:** Products load as you scroll; requires browser automation or API reverse-engineering
- **Lazy-loaded images:** Images only load when scrolled into view
- **Dynamic filtering:** Size/color availability loaded via AJAX after initial page load
- **Client-side rendering:** Entire product pages rendered in JavaScript
- **A/B testing:** Different users see different page layouts
- **Geo-blocking:** Different products/prices by region; requires regional proxies

### Data Normalization Challenges

- **Size formats:** S/M/L vs 0-16 vs 28-42 vs UK 8/10/12 vs EU 36/38/40
- **Color names:** "Midnight" vs "Navy" vs "Dark Blue" -- same color, different names
- **Material descriptions:** Inconsistent formatting across retailers
- **Price formats:** Different currencies, tax inclusion varies by country
- **Category taxonomies:** Every retailer has a different hierarchy

## 1. What is a SERP API?

A **SERP API** (Search Engine Results Page API) is a service that programmatically queries search engines (Google, Bing, Yahoo, etc.) and returns structured data (typically JSON) from the results pages. Instead of building and maintaining your own web scraping infrastructure -- dealing with proxies, CAPTCHAs, IP rotation, and anti-bot measures -- a SERP API handles all of that and delivers clean, parsed data.

**Core use cases:**
- SEO monitoring and rank tracking
- Competitive intelligence
- Price monitoring and e-commerce product research
- Market research and trend analysis
- Ad intelligence (tracking paid placements)
- Academic research

## Overview

Git worktrees create isolated workspaces sharing the same repository, allowing work on multiple branches simultaneously without switching.


**If found:** Use that directory. If both exist, `.worktrees` wins.

```bash
grep -i "worktree.*director" CLAUDE.md 2>/dev/null
```
