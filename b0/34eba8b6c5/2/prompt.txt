# Plan: Fix `start_session` 400 crash — invalid UUID for demo user

## Context

When clicking "Start Session" on the `/chat` page, the backend crashes with `400 Bad Request` from Neon. The frontend sends `user_id = "chat-demo-user"` (hardcoded at `chat/page.tsx:31`), but `users.id` is type `uuid` — Neon rejects the non-UUID string at query time. Even with a valid UUID, the session `INSERT` would fail if no matching user exists (FK constraint on `sessions.user_id`).

## Root Causes

1. `"chat-demo-user"` is not a valid UUID → Neon 400 on `SELECT ... FROM users WHERE id = $1`
2. No demo user record exists in `users` table → FK violation on `INSERT INTO sessions`
3. No error handling in socket handler (`main.py:54-60`) or orchestrator (`orchestrator.py:63-119`)

## Changes

### 1. `backend/migrations/006_create_demo_user.sql` — **Create**

Seed a demo user with a well-known UUID so the chat demo works out of the box.

```sql
INSERT INTO users (id, name, email)
VALUES ('00000000-0000-4000-a000-000000000001', 'Demo User', 'demo@mirrorless.local')
ON CONFLICT (id) DO NOTHING;
```

socket.on("session_error", (data: { error: string }) => {

  setMessages((prev) => [...prev, {

  }]);

Also add `socket.off("session_error")` in the cleanup return block.

### 3. `backend/main.py` — **Modify**

Add UUID validation + try/except in the `start_session` handler (lines 53-60):

```python
from uuid import UUID

def _is_valid_uuid(value: str) -> bool:
    try:
        UUID(value)
        return True
    except (ValueError, AttributeError):
        return False
```

Update `start_session` to validate UUID and catch errors, emitting `session_error` to the client via `to=sid` (not `room=user_id`, since the user may not have joined a room yet).

### 4. `backend/agent/orchestrator.py` — **Modify**

Wrap both DB blocks in `start_session` (lines 69-108) with try/except:
- **User data loading** (lines 69-76): catch exception → use empty defaults (`profile={}`, `purchases=[]`, etc.)
- **Session INSERT** (lines 100-108): catch exception → log and continue without DB persistence

This ensures the session still works (Claude conversation, Socket.io events) even if the user doesn't exist in DB.
