## Part A: Orchestrator Fixes (main agent)

### A1. Enforce API call limit — `backend/agent/orchestrator.py`

**Problem:** `SOFT_API_LIMIT = 20` (line 34) is defined but never checked. `_call_claude` ↔ `_handle_tool_calls` can recurse infinitely.

**Fix:** Add a guard at the top of `_call_claude()` (line 184):
```python
if session.api_calls >= SOFT_API_LIMIT:
    print(f"[mira] API limit reached for {session.user_id}")
    return
```

### A2. Concurrency guard — `backend/agent/orchestrator.py`

**Problem:** Silence timer fires during active Claude calls, causing parallel mutations to `conversation_history`.

**Fix:**
- Add `is_processing: bool = False` field to `SessionState` (line 51 area)
- Set `is_processing = True` at top of `handle_event()` (line 137), reset to `False` at end
- In `_silence_callback()` (line 403), check `session.is_processing` before calling `handle_event()`
- Wrap `handle_event` body in try/finally to always reset the flag

### A3. Error handling in `_call_claude` — `backend/agent/orchestrator.py`

**Problem:** No try/except around Claude API call (line 192). Failures leave conversation history in invalid state.

**Fix:** Wrap the streaming block in try/except. On failure, pop the last user message from history to keep it consistent, log the error.

### A4. Error handling in `_handle_tool_calls` — `backend/agent/orchestrator.py`

**Problem:** `execute_tool()` can raise (Serper `raise_for_status` at `tools.py:112`). Inconsistent with `search_gmail` which catches internally.

**Fix:** Wrap `execute_tool()` call in try/except inside `_handle_tool_calls()`. On failure, return an error dict as the tool result so Claude can handle it gracefully.

### A5. Wrap `_generate_summary` in try/except — `backend/agent/orchestrator.py`

**Problem:** `_generate_summary()` at line 281 is outside the try/except. Claude API failure prevents session cleanup.

**Fix:** Wrap in try/except, fall back to a generic summary string like `"Session ended before summary could be generated."`.

### A6. Disconnect cleanup + sid-to-user mapping — `backend/main.py`

**Problem:** Disconnect handler (line 51) does nothing. Orphaned sessions accumulate.

**Fix:**
- Add `_sid_to_user: dict[str, str] = {}` at module level
- In `join_room`, store the mapping: `_sid_to_user[sid] = user_id`
- In `disconnect`, look up `user_id = _sid_to_user.pop(sid, None)`, then call `mira.end_session(user_id)` if there's an active session
- Also add try/except around `mirror_event` handler (line 84-90)

### A7. Fix CORS mismatch — `backend/main.py`

**Problem:** `cors_allowed_origins=[]` on Socket.io server (line 12) rejects WebSocket upgrades in production.

**Fix:** Change to `cors_allowed_origins="*"` (or pass the frontend URL from env).

### A8. Session deduplication — `backend/agent/orchestrator.py`

**Problem:** Double `start_session` call orphans the first session's timers.

**Fix:** At top of `start_session()`, check if `user_id` already has an active session. If so, end it first:
```python
if user_id in self.sessions:
    await self.end_session(user_id)
```

## Part B: Prompt & Memory Fixes (main agent)

### B1. Empty purchase history fallback — `backend/agent/prompts.py`

**Problem:** If `_filter_fashion_purchases()` returns empty, Mira's opening (roast a specific purchase) has no data.

**Fix:** After the `## Recent Purchases` section in `build_system_prompt`, if `filtered_purchases` is empty, add:
```
## Recent Purchases
No purchase history available. Skip the purchase roast — instead, open by commenting on what you can see (their outfit, their vibe) and ask about their style preferences directly.
```

### B2. Include liked items from past sessions — `backend/agent/prompts.py` + `memory.py`

**Problem:** `load_past_sessions` returns full `outfit_data` dicts (which include `liked_items`), but `build_system_prompt` only uses `summary`.

**Fix:** In `build_system_prompt`, also extract `liked_items` from each past session dict:
```python
for session in session_history[-3:]:
    parts.append(f"- {session.get('summary', 'No summary')}")
    liked = session.get("liked_items", [])
    if liked:
        names = [item.get("title", "?") for item in liked[:3]]
        parts.append(f"  Liked: {', '.join(names)}")
```

### B3. Remove silence double-prompting — `backend/agent/prompts.py`

**Problem:** Personality says "be proactive if silent" AND the silence event sends an explicit command.

**Fix:** Remove the proactive silence instruction from `MIRA_PERSONALITY` line 14 (the silence timer event in `_event_to_message` already handles this at line 381-383).

### B4. Increase purchase fetch limit — `backend/agent/memory.py`

**Problem:** DB fetches 30 purchases, prompt filter discards many, leaving fewer than 20 fashion items.

**Fix:** Change `LIMIT 30` to `LIMIT 50` in `load_user_purchases` (line 102) to provide more buffer for the filter.

## Part C: DB Query Optimization (main agent)

### C1. Parallelize session-start DB calls — `backend/agent/orchestrator.py`

**Problem:** 5 sequential HTTP round trips to Neon at session start.

**Fix:** Use `asyncio.gather()`:
```python
profile, purchases, past_sessions, oauth_token = await asyncio.gather(
    load_user_profile(db, user_id),
    load_user_purchases(db, user_id),
    load_past_sessions(db, user_id),
    get_user_oauth_token(db, user_id),
)
```

### C2. Merge profile queries into JOIN — `backend/agent/memory.py`

**Problem:** `load_user_profile` makes 2 sequential HTTP calls (users + style_profiles).

**Fix:** Merge into one query:
```sql
SELECT u.id, u.name, u.email,
       sp.brands, sp.price_range, sp.style_tags, sp.size_info, sp.narrative_summary
FROM users u
LEFT JOIN style_profiles sp ON sp.user_id = u.id
WHERE u.id = $1
```

## Part E: Scraper Fix (separate parallel agent)

The scraper agent will fix the root causes of noisy purchase data in `backend/scraper/purchase_parser.py`:

1. **Tighten `_is_receipt()`**: Change from OR to AND logic — require BOTH sender match AND subject keyword match. Remove `"noreply@"` from `RECEIPT_SENDERS`.
2. **Fix subject-line fallback**: Require a price to be present before storing a subject as an item. Reject items where `item_name` has no fashion-related keywords.
3. **Fix `_categorize_item()`**: Add exclusion keywords so "short selling" doesn't become "bottoms" and "top picks" doesn't become "tops".
4. **Rebuild style profiles**: After fixing the parser, apply `_filter_fashion_purchases` logic at the `profile_builder.py` level too, so `build_style_profile()` only considers fashion items for brands/price_range.
