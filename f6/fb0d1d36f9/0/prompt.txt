# Plan: Fix MCP Server Database Connection Timeout

## Context

The Poke MCP server (`backend/mcp_server/server.py`) connects to Neon Postgres via asyncpg, which requires port 5432 to be reachable. On local dev environments with VPN/firewall restrictions, port 5432 is blocked, causing connection timeouts when tools like `get_past_sessions` are called.

The main FastAPI backend solves this with a dual-mode connection strategy in `backend/models/database.py`:
- **Production (Render)**: asyncpg pool via port 5432
- **Local dev**: Neon serverless HTTP API when port 5432 is blocked

The MCP server needs the same dual-mode fallback.

**Error:**
```
asyncpg.connection.py:2421 in connect
TimeoutError
```

## Proposed Solution

Add Neon HTTP fallback to the MCP server using the same pattern as the main backend.

### Implementation Approach: Try-Catch with Timeout

When `_get_pool()` is called:
1. **Try** to create an asyncpg pool with a short timeout (3 seconds)
2. **If successful** → use asyncpg for all queries (production path)
3. **If timeout or connection refused** → fall back to `NeonHTTPClient` (local dev path)
4. Cache the chosen client mode to avoid re-attempting asyncpg on every tool call

This approach optimizes for the production case (no penalty) while gracefully handling local dev.


Rewrite `_get_pool()` as `_get_db_client()`:
```python
async def _get_db_client() -> asyncpg.Pool | NeonHTTPClient:
    global _db_client, _db_mode

    if _db_client is not None:
        return _db_client

    # Try asyncpg first (production)
    log.info("Attempting asyncpg connection (port 5432)...")
    try:
        _db_client = await asyncio.wait_for(
            asyncpg.create_pool(
                DATABASE_URL,
                min_size=1,
                max_size=5,
                ssl="require",
            ),
            timeout=3.0  # Short timeout for fast failover
        )
        _db_mode = "asyncpg"
        log.info("Database connection: asyncpg (production mode)")
        return _db_client
    except (asyncio.TimeoutError, OSError, ConnectionRefusedError) as e:
        log.warning("asyncpg connection failed (%s), falling back to Neon HTTP", type(e).__name__)

    # Fallback to Neon HTTP (local dev)
    _db_client = NeonHTTPClient(DATABASE_URL)
    _db_mode = "http"
    log.info("Database connection: Neon HTTP (local dev mode)")
    return _db_client
```

c) **Update query logic to handle both client types:**

Create adapter functions that normalize the API:
```python
async def _fetch(client, query: str, *params):
    """Execute query and return all rows."""
    if isinstance(client, NeonHTTPClient):
        # Neon HTTP uses positional params as a list
        return await client.execute(query, list(params))
    else:
        # asyncpg uses pool.fetch
        async with client.acquire() as conn:
            return await conn.fetch(query, *params)

async def _fetchrow(client, query: str, *params):
    """Execute query and return first row."""
    if isinstance(client, NeonHTTPClient):
        rows = await client.execute(query, list(params))
        return rows[0] if rows else None
    else:
        async with client.acquire() as conn:
            return await conn.fetchrow(query, *params)

async def _execute(client, query: str, *params):
    """Execute query without returning rows."""
    if isinstance(client, NeonHTTPClient):
        await client.execute(query, list(params))
    else:
        async with client.acquire() as conn:
            await conn.execute(query, *params)
```

## Key Decisions

- **Why try-catch instead of pre-flight port check?** Pre-flight adds 3+ seconds to every cold start in production (unnecessary). Try-catch is instant in production (asyncpg succeeds immediately) and only takes 3s timeout penalty once on local dev boot.

- **Why 3-second timeout?** Long enough to succeed in production over slow networks, short enough to fail fast locally. asyncpg on Render typically connects in <200ms.

- **Why global singleton?** FastMCP tools are stateless functions. A singleton client avoids re-attempting asyncpg on every tool call after the initial fallback decision.

- **Row format differences?** asyncpg returns `Record` objects (dict-like), Neon HTTP returns plain dicts. Both support dict key access (`row["id"]`), so the adapter layer works transparently.
