# Plan: Add `take_photo` Tool for Mira

## Context

Mira currently receives one snapshot automatically at session start (`orchestrator.py:182-184`), but has no way to request another photo mid-conversation. The existing `request_snapshot` → `mirror_event` flow also has a bug where the frontend sends the wrong data format, so snapshots are silently dropped by `main.py`. This plan adds a `take_photo` tool Mira can invoke on demand and fixes the existing snapshot bug.

## Architecture

The core challenge: during tool execution, `is_processing = True` (orchestrator.py:199) blocks `handle_event()` from processing incoming events. So a snapshot response routed through `mirror_event` → `handle_event()` would deadlock. Solution: use a **separate Socket.io event channel** (`request_photo` → `photo_response`) that resolves an `asyncio.Future` directly, bypassing `handle_event` entirely.

```
Claude calls take_photo
  → orchestrator creates asyncio.Future, emits "request_photo"
  → frontend captures frame, emits "photo_response"
  → main.py handler resolves Future (no handle_event)
  → tool returns multimodal result with image
  → Claude sees the photo immediately
```

**C. Add `_take_photo()` method** on `MiraOrchestrator`:

```python
async def _take_photo(self, user_id: str, reason: str) -> dict:
    """Request a photo from the mirror and wait for the response."""
    if not self.sio:
        return {"error": "Socket.io not available"}

    loop = asyncio.get_running_loop()
    future: asyncio.Future[str] = loop.create_future()
    self._pending_photos[user_id] = future

    await self.sio.emit("request_photo", {"user_id": user_id}, room=user_id)

    try:
        image_base64 = await asyncio.wait_for(future, timeout=PHOTO_TIMEOUT_SECONDS)
    except asyncio.TimeoutError:
        return {"error": "Photo capture timed out — the mirror camera may not be ready"}
    finally:
        self._pending_photos.pop(user_id, None)

    if not image_base64:
        return {"error": "Photo capture returned empty data"}

    return {"success": True, "reason": reason, "image_base64": image_base64}
```

**D. Add `resolve_photo()` method** on `MiraOrchestrator` (called from `main.py`):

```python
def resolve_photo(self, user_id: str, image_base64: str) -> bool:
    """Resolve a pending photo Future. Returns True if resolved."""
    future = self._pending_photos.get(user_id)
    if future and not future.done():
        future.set_result(image_base64)
        return True
    return False
```

@sio.event
async def photo_response(sid, data):
    """Handle photo responses from the mirror (bypasses handle_event to avoid deadlock)."""
    user_id = data.get("user_id")
    image_base64 = data.get("image_base64")
    if not user_id or not image_base64:
        return
    resolved = mira.resolve_photo(user_id, image_base64)
    if not resolved:
        print(f"[socket] photo_response: no pending request for {user_id}")

## Edge Cases

- **Timeout** (5s): Tool returns error, Claude can naturally say "camera might not be ready"
- **Camera not ready**: Frontend returns nothing → Future times out → graceful error
- **Session ends mid-photo**: Future times out, `finally` block cleans up
- **No pending request**: `resolve_photo()` returns False, no crash
- **Image size**: JPEG 0.7 quality ≈ 50-200KB base64, well within Anthropic's 20MB limit
