# Plan: Fix give_recommendation Token Overflow

## Context

The Claude API is still failing with `213,932 tokens > 200,000 maximum` at turn ~12. The diagnostic logging we added reveals the **real culprit** is NOT base64 images (those are now properly stripped), but the `give_recommendation` tool returning **671,282 chars** (~168k tokens) of plain text:

```
[mira] Tool result for give_recommendation: raw=671,282 → stripped=671,282 chars
```

The stripping correctly shows `raw=stripped` — there are no data URLs to strip. This is pure text: 50 product listings with long Google Shopping URLs.

## Root Cause

Two compounding issues in `backend/agent/tools.py`:

1. **Too many items**: `execute_give_recommendation` returns 50 items (25 tops + 25 bottoms). Each formatted item is ~1000-2000 chars (dominated by long URLs). On cache hit, ALL ~90 raw items get formatted (no limit applied).

2. **Verbose format with unnecessary URLs**: `_format_clothing_for_claude()` includes `Link:` (Google Shopping redirect URL, 500-2000 chars) and `Image:` (CDN URL, 200-500 chars) per item. Claude doesn't need these — it only needs `product_id`, `title`, `image_url`, and `source`/`price` to call `display_product`.

3. **No insertion-time size cap in orchestrator**: `_handle_tool_calls()` stores the full tool result JSON in conversation history with no maximum size.

The result survives all compaction because it's in the last 2 messages (just added), and `_emergency_compact` / `_prepare_messages` only strip images/data URLs, not large strings.

## Files Modified

1. **`backend/agent/tools.py`** — reduce give_recommendation output
2. **`backend/agent/orchestrator.py`** — add insertion-time size cap + prepare-time cap

## Changes

### 1. Slim down `_format_clothing_for_claude()` (tools.py:737-767)

Remove `Link:` and `Rating:` lines from per-item format. Keep `Image:` (Claude needs image_url for display_product) but these are the shorter URLs. Remove `Link:` which contains the massive Google Shopping redirect URLs.

New format per item:
```
- **{title}** | {source} | {price}
  Image: {image_url} | ID: {product_id}
```

Reduces per-item output from ~1000-2000 chars to ~400-600 chars.

### 2. Limit items in `_format_clothing_for_claude()` (tools.py:737)

Add a hard cap of **10 items per category** (10 tops + 10 bottoms = 20 max) inside `_format_clothing_for_claude` itself, not just at the call site. This fixes the cache-hit path (line 693) which currently bypasses the `_select_diverse_items` limit.

```python
tops = tops[:10]
bottoms = bottoms[:10]
```

**Expected output size**: 20 items × 500 chars = ~10,000 chars. Well under limits.

### 3. Add `MAX_TOOL_RESULT_CHARS` cap at insertion time (orchestrator.py:~870)

After the two-pass data URL stripping, add a hard character cap:

```python
MAX_TOOL_RESULT_CHARS = 20_000  # ~5k tokens max per tool result in history
if len(stripped_json) > MAX_TOOL_RESULT_CHARS:
    stripped_json = stripped_json[:MAX_TOOL_RESULT_CHARS] + ' ...[truncated]'
    print(f"[mira] Tool result for {tool_use.name} truncated: {len(stripped_json):,} chars")
```

This is a safety net — with Change 1+2, no tool should exceed 20k chars. But if one does, it gets truncated rather than blowing up the context.

### 4. Add size cap in `_prepare_messages()` (orchestrator.py:~609)

In the tool_result sanitization path, after stripping data URLs, add a size cap:

```python
if isinstance(rc, str):
    cleaned = self._strip_data_urls_in_string(rc)
    if len(cleaned) > MAX_TOOL_RESULT_CHARS:
        cleaned = cleaned[:MAX_TOOL_RESULT_CHARS] + ' ...[truncated]'
    new_content.append({**block, "content": cleaned})
```

This catches any tool results that were stored before the insertion-time cap was added (e.g., from earlier in the conversation before a code deploy).
