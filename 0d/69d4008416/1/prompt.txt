# Plan: Admin Dashboard — Deepgram Sensitivity + Scrape Job Status

## Context

Two admin dashboard features: (1) expose Deepgram's `endpointing` parameter as a "sensitivity" slider — controls ms of silence before speech is finalized at the phrase level, complementing the existing `utterance_end_ms` which operates at the full-utterance level; (2) show email scraping job status with phase, purchase count, and errors — currently scrape jobs are fire-and-forget with zero admin visibility.

## Feature 1: Deepgram Sensitivity (endpointing slider)

The existing STT config pipeline (`_stt_config` dict → POST `/admin/stt-config` → `stt_config_updated` socket → hook hot-reload) already handles 3 params. We add a 4th: `endpointing` (10–500ms, default 10ms). Lower = more sensitive/faster finalization, higher = waits longer for continued speech.

### Changes

**`backend/routers/admin.py`**
- Add `"endpointing": 10` to `_stt_config` dict (line 15)
- Add `endpointing: Optional[int] = Field(None, ge=10, le=500)` to `STTConfigUpdate` (line 22)
- Add `if body.endpointing is not None:` update block in `update_stt_config()` (line 43)

**`frontend/src/hooks/useDeepgramSTT.ts`**
- Add `endpointing: number` to `DeepgramSTTConfig` interface (line 8)
- Add `endpointing: 10` to `DEFAULT_CONFIG` (line 14)
- Add `endpointing: String(config.endpointing)` to `buildDeepgramUrl()` params (line 21)
- Add `activeConfig.endpointing` to hot-reload `useEffect` dependency array (line 213)

**`frontend/src/lib/api.ts`**
- Add `endpointing: number` to `STTConfig` interface (line 222)

**`frontend/src/app/admin/page.tsx`**
- Add `localEndpointing` state in `STTSettings` (same pattern as existing `localUtterance`)
- Add slider: min=10, max=500, step=10, save on mouseUp
- Label: "Sensitivity (Endpointing): {value}ms"
- Helper text: "10ms (fast/sensitive)" ↔ "500ms (slow/patient)"
- Place between Utterance End and Model dropdown

## Feature 2: Scrape Job Status

In-memory `OrderedDict` tracking (max 50 jobs, evict oldest completed). No DB table — scrape jobs are transient, server-restart loss is fine for admin monitoring.

### Changes

**`backend/scraper/routes.py`** — Add job tracking module-level:
- `_scrape_jobs: OrderedDict[str, dict]` with helper functions:
  - `_update_scrape_job(user_id, **updates)` — update phase/purchases
  - `_complete_scrape_job(user_id, status, error=None)` — mark done/failed
  - `get_scrape_jobs()` — return list (newest first) for admin endpoint
- In `start_scrape()`: look up user name from DB, create job entry before `create_task()`
- In `_background_scrape()`: call `_update_scrape_job()` at each phase transition (searching → parsing → profiling → complete), increment `purchases_found` in `on_email` callback, call `_complete_scrape_job()` on success/failure

**`backend/routers/admin.py`**
- Add `GET /admin/scrape-jobs` endpoint that imports and calls `scraper.routes.get_scrape_jobs()`

**`frontend/src/lib/api.ts`**
- Add `ScrapeJob` interface: `{ user_id, user_name, status, phase, purchases_found, started_at, completed_at, error }`
- Add `getScrapeJobs()` function calling `GET /admin/scrape-jobs`

**`frontend/src/app/admin/page.tsx`**
- Add `scrapeJobs` state, fetch in `refresh()` via `Promise.all`
- Add "Scrape Jobs" section after STT Settings
- `ScrapeJobRow` component: user name, status badge (running=blue, completed=green, failed=red), phase label, purchase count, elapsed time, error text
- Show most recent 10 jobs
