## Step 1: Migration — `backend/migrations/007_create_calendar_events.sql`

New table mirroring the `purchases` pattern (CASCADE delete, unique constraint for upserts):

```sql
CREATE TABLE calendar_events (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  google_event_id text NOT NULL,
  title           text NOT NULL,
  start_time      timestamptz NOT NULL,
  end_time        timestamptz,
  location        text,
  description     text,          -- truncated to 500 chars
  attendee_count  integer DEFAULT 0,  -- non-self attendees only (no PII)
  is_all_day      boolean DEFAULT false,
  status          text,          -- confirmed/tentative/cancelled
  scraped_at      timestamptz NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_calendar_events_user_google_id ON calendar_events(user_id, google_event_id);
CREATE INDEX idx_calendar_events_user_start ON calendar_events(user_id, start_time);
```

- Uses `ON CONFLICT DO UPDATE` (not `DO NOTHING` like purchases) because calendar events change after creation
- `attendee_count` instead of a full attendees list — avoids PII, Mira only needs "dinner for 2" vs "conference with 200"

## Step 2: Calendar Fetch Module — `backend/scraper/calendar_fetch.py`

Mirrors `gmail_auth.py:46-57` (`build_gmail_service`) pattern:

- `build_calendar_service(token_data)` — same `Credentials` + `build()` pattern, just `("calendar", "v3")`
- `fetch_events(service, days_back=7, days_forward=14, max_results=50)` — calls Calendar API v3 `events().list()` with `singleEvents=True` (expands recurring), `orderBy="startTime"`
- `_normalize_event(item)` — extracts title, start/end times, location, description (truncated 500 chars), attendee_count (excluding self), is_all_day, status

Key: `singleEvents=True` is critical — it expands "Weekly team standup" into individual instances so Mira sees the actual upcoming one.

## Step 4: Agent Memory — Add to `backend/agent/memory.py`

Two new functions:

- `load_calendar_events(db, user_id)` — queries DB for events past 7 days through next 14 days, excludes cancelled. Same pattern as `load_user_purchases()`.
- `refresh_calendar_events(db, user_id, token_data)` — live-fetches from Google Calendar API, stores in DB, returns events. Uses `ThreadPoolExecutor` + `run_in_executor` (same pattern as `pipeline.py`) since `googleapiclient` is synchronous. Runs at session start for freshness.
