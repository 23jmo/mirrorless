# Plan: Migrate from Mira to Poke as Agent Brain

## Context

Mirrorless currently uses **Mira**, a custom Claude-based agent orchestrator (`backend/agent/`), as the AI brain for the smart mirror shopping assistant. We're pivoting to **Poke** (by Interaction Company) — an AI assistant in iMessage/Telegram/SMS that acts as an MCP host.

**The pivot**: Instead of running our own AI brain, we expose our shopping tools via an MCP server. Users install a Poke **recipe** that auto-connects our tools to their Poke agent. Poke already has complete contextual awareness over the user (Gmail, calendar, contacts) via its own integrations — so it knows what the user bought, what events are coming up, and what their style is. Our tools add the ability to **search for new clothes** and **display items on their physical mirror**.

**What changes:**
- Poke becomes the brain (replaces orchestrator, prompts, Claude API calls, session management)
- Poke handles user data (replaces our Gmail/calendar scraping pipeline)
- We build a lightweight MCP server with 5 tools

**What stays:**
- Serper clothing search API (our unique value-add)
- Mirror display via Socket.io (present_items + send_to_mirror push to physical mirror)
- Frontend mirror display app
- Existing FastAPI backend (simplified — serves Socket.io + mirror bridge)

### Streaming Limitation

MCP does NOT support streaming tool results (confirmed by FastMCP maintainer — "Streamable HTTP" is misleadingly named). Tools return a single complete payload. Poke is message-based (iMessage/SMS), so responses are complete messages, not streams. Our `send_to_mirror` tool receives complete text blocks. For TTS, the mirror can speak each block as it arrives — each tool call is a "speech turn." This is sufficient for conversational UX.

## Mirror Linking: Permanent Mirror ID

Each physical mirror has a **permanent ID** configured at setup (e.g., `"MIRROR-A1"`). This is displayed on the mirror screen and printed on the device. The user tells Poke their mirror ID once during recipe onboarding — Poke remembers it across sessions (Poke's 3-layer memory: conversation logs, agent logs, summarized context).

**Why permanent mirror_id (not temporary codes or poke_id):**
- No user registration or DB lookup needed — fully stateless
- Permanent — doesn't expire or change on restart
- User can re-enter it anytime by just telling Poke in conversation
- Mirror joins Socket.io room `mirror:<mirror_id>` at startup
- MCP server resolves `mirror_id` → HTTP POST to backend → emit to that room

**Future upgrade path**: For multi-user production, add poke_id-based linking (the `poke_id` column already exists in the `users` table). This would let us map Poke accounts to mirrors persistently in the DB.

## Phase 2: Add `present_items` + `send_to_mirror` with Mirror Bridge

**Goal:** Poke can push clothing items AND text/speech to the physical mirror display.

### Architecture

```
Poke → MCP server (present_items / send_to_mirror)
         → HTTP POST to FastAPI backend (/api/mirror/*)
              → Socket.io emit to room "mirror:<mirror_id>"
                   → Mirror display updates (items / text + TTS)
```

The MCP server does NOT connect to Socket.io directly — it calls HTTP endpoints on the existing FastAPI backend, which handles Socket.io emission. This keeps the MCP server stateless.

### Five MCP tools total after this phase

| Tool | Purpose | Socket.io Event |
|------|---------|----------------|
| `search_clothing` | Search Serper API (Phase 1) | None (returns to Poke only) |
| `present_items` | Show clothing cards on mirror | `tool_result` with `type: "clothing_results"` |
| `send_to_mirror` | Show text on mirror (+ TTS) | `mirror_text` with `{text}` |
| `get_past_sessions` | Query previous shopping sessions | None (returns to Poke only) |
| `save_session` | Save current session summary + reactions | None (writes to DB) |

### Files to create

**`backend/mcp/server.py`** — Add both tools

`present_items(mirror_id: str, items: list[dict])`:
- Each item must have: title, price, image_url, link, source
- Calls `POST {BACKEND_URL}/api/mirror/present` with the payload
- Returns success/failure to Poke

`send_to_mirror(mirror_id: str, text: str)`:
- Sends Poke's commentary/speech text to the mirror display
- Calls `POST {BACKEND_URL}/api/mirror/text` with `{mirror_id, text}`
- Mirror displays the text and can run TTS on it
- Returns success/failure to Poke
- This replaces the old `mira_speech` socket event

`get_past_sessions(mirror_id: str, limit: int = 5)`:
- Returns previous shopping sessions for this mirror
- Each session includes: date, items shown, reactions (liked/disliked/skipped), summary
- Calls `GET {BACKEND_URL}/api/mirror/sessions/{mirror_id}?limit=5`
- Poke uses this to recall "last time we looked at sneakers you liked the Nike ones"
- Reuses existing `sessions` + `session_outfits` tables (with new `mirror_id` column)

`save_session(mirror_id: str, summary: str, items_shown: list[dict], reactions: dict)`:
- Saves the current shopping session when the interaction wraps up
- Calls `POST {BACKEND_URL}/api/mirror/sessions` with session data
- Each item in `items_shown` can have a reaction: "liked", "disliked", "skipped"
- `reactions` is an aggregate: `{likes: 2, dislikes: 1, items_shown: 5}`
- `summary` is a brief text recap Poke generates (e.g., "Looked at date night outfits, loved the navy blazer")
- Poke should call this at the end of each shopping conversation

**`backend/routers/mirror.py`** — New REST endpoints on existing FastAPI backend
- `POST /api/mirror/present` — Receives `{mirror_id, items}`, emits `tool_result` to Socket.io room
- `POST /api/mirror/text` — Receives `{mirror_id, text}`, emits `mirror_text` to Socket.io room
- `GET /api/mirror/sessions/{mirror_id}` — Returns past sessions with items and reactions
- `POST /api/mirror/sessions` — Saves a new session record
- `GET /api/mirror/status/{mirror_id}` — Check if a mirror is connected (optional)
