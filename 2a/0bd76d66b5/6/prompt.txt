# Plan: Fix admin dashboard 500 errors — SQL column/table mismatches

## Context

The admin dashboard (`/admin/session` and `/admin/stats`) crashes with 500 errors because the SQL queries reference columns and tables that don't exist in the database schema. The queries were written speculatively during the initial admin dashboard implementation without matching the actual migration in `001_initial_schema.sql`.

**Errors observed:**
- `column "api_call_count" does not exist` — on `/admin/session`
- `column "created_at" does not exist` — on `/admin/stats`

## Root Cause

The `sessions` table (defined in `backend/migrations/001_initial_schema.sql:36-42`) has:
```
id, user_id, started_at, ended_at, status
```

The admin queries incorrectly reference:
1. `api_call_count` — never existed; api_calls are tracked in-memory by the orchestrator (`SessionState.api_calls`), not in the DB
2. `created_at` — should be `started_at`
3. `outfit_items` table — should be `session_outfits` (the actual table name per migration)

## Fix

**File: `backend/routers/admin.py`**

### `/admin/session` endpoint (lines 60-67)

| Line | Bug | Fix |
|------|-----|-----|
| 62 | `SELECT id, api_call_count` | `SELECT id` (remove `api_call_count`) |
| 64 | `ORDER BY created_at DESC` | `ORDER BY started_at DESC` |
| 73 | `api_calls = session_rows[0].get("api_call_count", 0)` | `api_calls = 0` (not available from DB) |
| 80 | `FROM outfit_items` | `FROM session_outfits` |
| 86 | `FROM outfit_items` | `FROM session_outfits` |

### `/admin/stats` endpoint (lines 115-138)

| Line | Bug | Fix |
|------|-----|-----|
| 119 | `ended_at - created_at` | `ended_at - started_at` |
| 121 | `created_at::date = CURRENT_DATE` | `started_at::date = CURRENT_DATE` |
| 127 | `FROM outfit_items oi` | `FROM session_outfits oi` |
| 129 | `s.created_at::date` | `s.started_at::date` |
| 133 | `FROM outfit_items oi` | `FROM session_outfits oi` |
| 135 | `s.created_at::date` | `s.started_at::date` |

### `api_calls` field

The `api_calls` count is tracked in the orchestrator's in-memory `SessionState`, not persisted to the DB. The admin endpoint will return `0` for now. The frontend type (`AdminSessionInfo.api_calls`) stays — it just gets `0` until we add DB persistence (out of scope).
