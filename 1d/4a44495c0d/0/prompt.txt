# Plan: Poke-Style Tiered Purchase Memory for Mira

## Context

Mira's personality depends on referencing specific purchases ("You spent $85 on a COS shirt..."), but currently only 20 items make it into the system prompt. Users with 100+ purchases have 80% of their history invisible to Mira.

**Inspiration**: [Poke by The Interaction Company](https://www.shloked.com/writing/openpoke) uses tiered memory — recent data at full fidelity, older data progressively compressed, plus on-demand retrieval into the full archive. We'll adapt this pattern for Mira's purchase data.

**Current bottleneck**: DB fetches 50 → filter keeps ~25 → hard cap at 20 → ~500 tokens used out of 200K available (<0.3% of context).

---

## Changes (4 components)

### 1. Purchase Summary Stats — `backend/agent/memory.py`

Add `load_purchase_statistics()` that computes aggregates over the **full** purchase history via SQL:
- Total count + total spend + avg/min/max price
- Top 10 brands by frequency (with spend per brand)
- Category breakdown
- Monthly spending trend (last 6 months)

This runs in `asyncio.gather` alongside the existing loads — one extra Neon round-trip.

**Reuse**: Import `_NON_FASHION_BRANDS` from `prompts.py` and generate a SQL filter clause from it, so the exclusion list stays in one place.

Also increase `load_user_purchases` LIMIT from 50 → 200 (more data for tiering).

### 2. Tiered Prompt Display — `backend/agent/prompts.py`

Replace the flat `[:20]` list with `_build_tiered_purchases()`:

| Tier | Window | Format | Cap |
|------|--------|--------|-----|
| Recent | Last 30 days | Full: brand, item, price, date, category | 25 items |
| Older | 30–90 days | Compact: brand + item + price | 30 items |
| Historical | 90+ days | Brand counts only: "Nike x4, Zara x3" | All brands |

Preceded by the aggregate stats section from Component 1. Followed by a note telling Claude about the `search_purchases` tool.

Add `purchase_stats` parameter to `build_system_prompt()`.

**Token budget**: ~1,500 tokens (vs current ~500). Still <1% of 200K context.

### 3. `search_purchases` Tool — `backend/agent/tools.py`

New Claude tool for on-demand DB retrieval. This is the Poke-style "archive as retrieval infrastructure" pattern — Claude can query ANY purchase ever scraped.

**Input schema**:
- `query` (string) — free-text search against brand/item_name via ILIKE
- `brand` (string) — exact brand filter
- `category` (string) — exact category filter
- `min_price` / `max_price` (number) — price range
- `date_from` / `date_to` (string) — date range
- `limit` (int, default 20, max 50)

**Output**: Matching purchases + `total_matching` count.

**SQL**: Dynamic WHERE clause with parameterized inputs. Uses existing `NeonHTTPClient`. Wrap in try/except matching `_search_gmail` pattern.

Add `NeonHTTPClient` import to tools.py. Add to `TOOL_DEFINITIONS` list and `execute_tool()` dispatch.

### 4. Enhanced Narrative — `backend/scraper/profile_builder.py`

Replace the 1-sentence narrative with `_build_enhanced_narrative()` that includes:
- Brand loyalty pattern (dominant brand % or diverse shopper)
- Shopping frequency (items/month)
- Splurge vs. basics pattern

This is stored in `style_profiles.narrative_summary` and already flows into the system prompt at `prompts.py:127-128`.
